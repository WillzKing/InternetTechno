1. GraphQL vs REST (концепт)
REST: Один эндпоинт — фиксированный набор данных. Клиент качает лишнее (оверфетчинг) или недокачивает (андерфетчинг).
GraphQL: Один эндпоинт. Клиент запрашивает только те поля, которые нужны. Контракт данных диктует клиент.

2. SDL (Schema Definition Language)
Язык описания типов. Строгий контракт между фронтом и бэком.
type User { id: ID! name: String }

3. Query vs Mutation
Query: Чтение. Параллельное выполнение.
Mutation: Изменение. Последовательное выполнение (одна за другой, чтобы избежать race conditions).

4. Резолвер (Resolver)
Функция, которая знает, откуда взять данные для конкретного поля в схеме (БД, другой API, вычисление).
Кто вызывает: GraphQL-движок на сервере в процессе парсинга входящего запроса.

5. Проблема N+1 и DataLoader
Проблема: Получить 10 постов -> для каждого сбегать в БД за автором (1 запрос за постами + 10 за авторами = 11).
DataLoader: Собирает ID авторов в кучу и делает один запрос WHERE id IN (...). Это группировка запросов в рамках одного тика Event Loop.

6. Сложность HTTP-кэширования
Почти все запросы в GraphQL идут через POST на один и тот же URL (/graphql).
Тело запроса меняется, а URL нет. HTTP-кэш (CDN, браузер) не умеет анализировать тело, поэтому не может отличить {user{name}} от {user{email}}. Нужно кэшировать на уровне клиентского Apollo.
